1 Back To The Books

Toward the end of my PhD I was struggling to learn new skills such as data analysis and programming, it was at that time that I first rediscovered books (duh!) and then discovered a very special kind of them, known as open books or open source books. This kind of tools helped me in my professional development, so I feel particularly attached to them.

By now I have found three types of open source books:

  • Books that are simply distributed by the author on their own website,
  • Books that are written through wikis, known as wikibooks,
  • Books that are written through Github.

I will discuss the last type because Github is powerful tool to write books that fits perfectly to academics.

A disclaimer first: I come here as a beginner. My area of expertise is biology and I have never written a book about it, but I would really like to do so. Meanwhile I collected the necessary tools and some idea on why we should do it. And I would like to share them with you.

2 Tools For Writing Open Books on Github

It all revolves around two tools: Markdown and, you have guessed it, Github.

2.1 Markdown

Markdown is a simple markup language. That is a language for annotating plain text; so that we know which part is the title, which part should be bold or italic, which part makes up paragraph, etc…

The most widely used markup languages are HTML and LATEX. Compared to them markdown is less versatile, but incredibly simple and easy to use.

For example, in markdown:

# This is a title

## This is a subtitle

This is a paragraph.

This text is **bold**, and this text is *italic*

This is a bullet list:

- item1
- item2

[This is a link](to/your/favourite/URL)

etc…

A big chunk of the web is written in Markdown; for example, this blog and many others.

Once you have written markdown text what can you do with it?

Markdown is a universal source that can be easily converted into nicely formatted docx and pdf, but also into HTML and epub/mobi. So, through pandoc you can convert any markdown file into any of those formats, no edits required.

Since markdown is plain text, you can read it on any computer and any platform, you just need a text editor. It is not even worth mentioning that pandoc is open source.

Also, since markdown is plain text, it can be tracked with Git and hosted on Github. (if you already know what they are, skip the next section)

2.2 Git and Github

Git is a version control software. It was designed for programmers that need to keep track of what they write and delete while they develop their programs; and also to collaborate while the do this. I am not the right person to provide details of what Git is. Generally Git is a tool that is somehow complementary to a backup, it keeps track of specific files on a project-wise schedule.

By keeping track of modification to a file, Git assigns exact authorship to each line of such file. So Git, plus its online hub Github allows you to publish, update, and keep track of modification and authorship of files online and in an open way, allowing open but controlled collaboration.

Since the source code of programs is indeed text, Git works perfectly also to write books (and blog, websites, manual etc…), as long as they are written in plain text formats, such as markdown.

2.3 Putting it together: Books, authorship and contributions

A combination of Markdown (universal source) and Github (publish, author, track and collaborate) is good combination for publishing open source books, many academics such as Hadley Wickham are already using it.

2.3.1 Authorship and contributions

I would like to take one of Hadley Wickham’s books as example to discuss authorship and contributions, because as academics we want to get credit for our work. As you can see in the book’s source Github page, Github tracks authorship of every line of the book. Hadley Wickham authored most of the book itself, but many other people provided small contribution. For every contributor, you can tell exactly what they wrote or deleted in the various files that compose the source for the book. What you can’t see is that each of these contribution had to be approved from the main author.

So, a combination of markdown and Github allows you to self publish your own books. This would be already powerful by itself. Moreover this system gains extra strength from Github’s authoring tools: anybody can propose changes / contributions to your book, but every contribution must be approved by you (or by any of a set of authors that you provided with such privilege).

be aware, since your book is now an open source project, it can be forked

2.4 Bonus: Bookdown

An experienced user can rely on its own skills to convert all its markdown chapters files into a nicely formatted HTML book, the good news is that we don’t have to do this by ourself: Bookdown takes care of rendering your book, partly through Pandoc, and more.

Bookdown’s manual is written in Bookdown itself, so you can peek how the rendering looks like. Bookdown is an R package. Therefore it is most widely known to the R community. Indeed most of the book written in Bookdown are about R. But they don’t need to: Bookdown is a fantastic tool and using it requires very little to no knowledge of R.

Another powerful tool is Gitbook, it was used to author this collection of technical books

3 Publishing

3.1 Own website

Markdown and Pandoc provide an HTML output, so your book can be easily integrated into any website. The book can be uploaded manually or automatically. It looks like travis-ci can build your book automatically from Github and push it to your website; but I have not checked the details and setting this up might not be easy.

In this way, you keep the copyright and you decide how to license your book (it is reasonable to use Github together with open source or share-alike licenses).

3.2 Rstudio connect, through Bookdown

Bookdown provides an integrated system publish your book to through Rstudio connect to this website. This is detailed on the “Publishing” chapter of bookdown’s manual together with other extensive information on publishing.

3.3 Publishers

It is also possible to sell open source books through publishers. Among those Leanpub does a good job, I personally purchased books in there. Leanpub accepts to sell open source books written in markdown, those books can also be uploaded directly from Github. Their publishing process is fast; indeed they accept to distribute your book in real time while you write it, providing updates at anytime to the customers.

Most of the books on Leanpub are about programming, but it does not have to be like this, we could publish there books on biology, or about any of your fields of interest.

If your open source book is incredibly good, you can also sell printed versions through traditional channels. Indeed Hadley Wickham sells his books through Chapman & Hall.

4 Impact

Again, a small disclaimer: I have not written nor published an open source book yet; so i put together this section from observation and experience while studying and working in academia in the last years.

4.1 Paywall

We scientists often complain about how people don’t trust and don’t make use of the scientific knowledge that we produce. But how much scientific knowledge is locked behind a paywall that has questionable reasons to exist? Writing open source books, at least while we are funded with public money, could have an ethical side.

What about the practical side? We academics do not earn huge amounts of money for writing books that are often sold at high prices. If we would distribute open source books instead, we would not lose much. Though we could gain impact: an open source book can be accessed worldwide, and it could be also easily translated.

Beside, remember that open source books can always be sold through Leanpub.

4.2 Updates and crowdsourcing

Imagine the impact of a book that can be freely:

  • accessed online,
  • downloaded on a e-reader,
  • printed,

And that can also be updated in real time. When we specialize we sometimes feel that books gets useless because they get obsolete too quickly. Well, open source books do not.

Moreover they are less of a burden on the author, because readers can contribute to the book and improve it, from fixing typos to writing whole sections.

For example, the author of Bookdown, (Yihui Xie), is also writing an R package to make website (that by the way, I am using to make this blog). The manual for the new package is being written with Bookdown and it has already many contributions on Github.

What better peer review than the readers themselves?

Did I already mentioned that engaging in these activities might look good on your CV?

5 Wrapping it up and what’s next

A combination of Markdown and Github is a powerful combination for publishing open source books; engaging in this activity, expecially for academics, could have great impact. Moreover, packages such as Bookdown make it easy to compile very nicely formatted HTML books with little effort.

Using this system we could publish educational book or also short and very specific high level reference books on the topics / area that we are most experienced in. Aiming for example at University students and people that want to learn these topics.

I don’t know if I am an expert in any of my area of interest to write a an open source book on it. But it sounds like a good long term project and I definitely look forward to do it. the tools are all there.
